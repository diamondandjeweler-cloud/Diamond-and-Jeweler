1. Mission & Vision
Mission

To create a fair, intelligent, and human‑centric recruitment ecosystem where every talent finds a leader who truly values them, and every leader discovers the talent that elevates their team — guided by proven insights, market honesty, and active advocacy.

Vision

A world where job matching is no longer a transaction of resumes and job descriptions, but a trusted partnership built on deep compatibility, mutual respect, and shared success — unlocking human potential for millions across Southeast Asia and beyond.

2. Executive Summary
BoLe is a dual‑sided recruitment platform that flips the traditional model:

Talents complete a system interview (20 questions + 20 preference ratings) and receive exactly 3 job offers at a time.

Hiring Managers (not HR) define role expectations and receive exactly 3 candidates at a time.

Hidden matching engine uses 16 years of proprietary feng shui / life‑chart data (DOB‑based) – weighted 30% – combined with behavioural tags (70%).

Market rate balancing educates both sides when expectations deviate.

“Fight the interview” active advocacy post‑match (Phase 2).

Progressive matching speed – initial wait periods shorten as database grows.

Launch: Malaysia first, with a new dedicated domain bole.my.
Timeline: 28 days to public launch (including legal review and PDPA compliance).

6. Database Schema (with pgsodium encryption)
Full migration script – includes all tables, pgsodium setup, and helper functions.
Only key excerpts shown here; complete script is in the deliverable package.

sql
-- Enable extensions
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS "pgcrypto";
CREATE EXTENSION IF NOT EXISTS "pg_cron";
CREATE EXTENSION IF NOT EXISTS "pgsodium";

-- Create a key for DOB encryption in Vault
SELECT pgsodium.create_key('dob_key');

-- Helper function to check admin role (SECURITY DEFINER + GRANT)
CREATE OR REPLACE FUNCTION auth.is_admin()
RETURNS boolean
LANGUAGE sql
SECURITY DEFINER
STABLE
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.profiles
    WHERE id = auth.uid() AND role = 'admin'
  );
$$;
GRANT EXECUTE ON FUNCTION auth.is_admin() TO authenticated;

-- Profiles table (no DOB here)
CREATE TABLE public.profiles (
  id uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email text UNIQUE NOT NULL,
  full_name text NOT NULL,
  phone text,
  role text NOT NULL CHECK (role IN ('talent', 'hiring_manager', 'hr_admin', 'admin')),
  consents jsonb DEFAULT '{}',
  is_banned boolean DEFAULT false,
  ghost_score integer DEFAULT 0,
  onboarding_complete boolean DEFAULT false,
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);

-- Companies
CREATE TABLE public.companies (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  registration_number text UNIQUE NOT NULL,
  business_license_url text,
  website text,
  size text,
  industry text,
  primary_hr_email text NOT NULL,
  verified boolean DEFAULT false,
  created_at timestamptz DEFAULT now()
);

-- Hiring Managers (encrypted DOB using pgsodium)
CREATE TABLE public.hiring_managers (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  profile_id uuid REFERENCES public.profiles(id) NOT NULL,
  company_id uuid REFERENCES public.companies(id) NOT NULL,

8. Row Level Security (RLS) – Complete Policies
All tables have RLS enabled. Policies ensure:

Users can only access their own data (profiles, talents, hiring_managers).

HR admins can read company data and matches for their company.

Hiring managers can read roles for their company and matches for their roles.

Talents can only see their own matches (and only after they are generated).

Admin has full access via auth.is_admin().

Insert policies are properly restricted (e.g., hiring_managers can only be inserted by the invite-hm Edge Function or admin).

Full policy set is in the migration file – includes all tables.

10. Edge Functions (JWT‑verified, no open endpoints)
All Edge Functions verify the JWT and role before processing.

match-generate – called by database webhook after role insertion. Verifies caller is service role or admin. Generates 3 matches, dedupes, respects refresh limit, stores internal_reasoning.

match-expire – called by pg_cron every 6h. Flips expired matches, triggers replacement if within refresh limit.

notify – called only by other Edge Functions (service role). Sends email via Resend + stores in‑app notification.

invite-hm – called by HR dashboard (authenticated HR only). Uses supabase.auth.admin.inviteUserByEmail() to send magic link.

cold-start – called by admin only. Manually pairs talent with role.

No function is deployed with --no-verify-jwt – all enforce authentication.

11. React Frontend – Fixed Architecture
Auth listener – useSession hook uses onAuthStateChange and getSession correctly, with Zustand store.

Hooks at top level – No conditional hooks. Separate components for each step (e.g., <UploadStep />, <InterviewStep />, <PreferencesStep />) to avoid React rule violations.

Role‑based routing – App.tsx wraps routes in BrowserRouter. Onboarding pages are separate routes, not conditional renders inside App.

Error boundaries – Wrap each dashboard with error boundary.

Empty states – Show friendly messages when no matches.

Loading skeletons – Use skeleton loaders for async data.

12. Onboarding Flows (No Hook Violations)
Talent Onboarding – Three distinct components:

TalentUploadStep – IC + resume upload.

TalentInterviewStep – 20 questions, one per screen (state managed at parent level, no hooks inside conditional).

TalentPreferencesStep – 20 aspects rating.

Parent component TalentOnboarding uses a step state and renders the appropriate component, passing callbacks.

Hiring Manager Onboarding – Similar decomposition:

HMConsentStep – DOB input + consent.

HMLeadershipStep – 10 questions, each as separate sub‑component.

DOB is captured once and encrypted using pgsodium via Edge Function call (not client‑side).

13. Match State Machine
States and transitions:

text
generated → viewed → accepted_by_talent → invited_by_manager → hr_scheduling → interview_scheduled → interview_completed → offer_made → hired
                                                                                                               ↓
                                                                                                        offer_declined
generated → viewed → declined_by_talent
generated → viewed → invited_by_manager (if manager invites first) → ...
Expiry after 5 days of no action (configurable).
State transitions are enforced in Edge Functions, not client‑side.

16. Anti‑Ghosting & Expiry Logic
Expiry: match-expire Edge Function (cron every 6h) sets status = expired for matches older than match_expiry_days (default 5).

Replacement: If role still active and refresh count < limit, call match-generate to add a new match (excluding expired talent).

Ghosting detection: If a user has 3 consecutive expired matches without any accept/decline, increment ghost_score. When ghost_score >= 3, reduce match priority (admin can override).

17. Cold Start Strategy
Early stage (<500 talents):

Manual seeding: Admin can use cold‑start tool to select talents for a role. Creates matches directly.

Fallback message: If a role has <3 eligible talents, show: “Not enough talents yet. We are growing our pool. You will be notified when we have 3 candidates. Estimated wait: 14 days.”

Auto switch: Once total talents > 500, disable manual seeding and rely on algorithm.

18. Role Posting & Market Rate Warning
When a hiring manager posts a role:

System fetches market rate from market_rate_cache based on job title, location, experience level.

If offered salary is below market median, show warning:

“The market rate for this role is RM X–Y. Your offer is below this range. Consider adjusting salary or experience level to attract quality candidates.”

Employer can proceed anyway or edit.

The warning is in‑app only (no email).

19. “Fight the Interview” (Phase 2 – Post‑MVP)
After a match is accepted, BoLe will actively advocate:

For talent: Insider briefing on HM style, mock interview, salary negotiation.

For HM: Candidate coaching, feedback templates, offer assistance.

Implementation deferred to Phase 2. In Phase 1, after acceptance, the system simply connects the parties.

20. Legal & PDPA Compliance
Documents required before public launch:

Privacy Notice – explains DOB encryption, retention (30 days after account deletion), data subject rights.

Terms of Service – user responsibilities, dispute resolution, liability.

Consent checkboxes – DOB, video (Phase 2), market data comparison – stored in profiles.consents.

Data Subject Request (DSR) workflow – email to privacy@bole.my triggers manual process (admin tool to export/delete data).

Data retention cron – deletes soft‑deleted user records and related encrypted fields after 30 days.

IC storage retention – IC files deleted 30 days after account deletion.

DOB encryption commitment: pgsodium with Vault key, not reversible by anyone except via Supabase admin with access to Vault.

Legal review required: A Malaysian PDPA lawyer must approve the privacy notice and ToS before public launch.
